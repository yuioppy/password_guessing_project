# password_guessing_project
# Отчёт по исследованию влияния эволюции промптов на качество модели PassLLM

# 0. Ноутбук следует запускать на kaggle, так как все исходные данные ссылаются на /kaggle/input/competitions/

## 1. Цель работы

Целью данной работы было исследовать, как изменение текстовой инструкции, подаваемой на вход языковой модели, влияет на качество решения задачи targeted password guessing, то есть задачи предсказания нового пользовательского пароля по известному старому паролю. В качестве базовой модели использовалась архитектура семейства Qwen, дополнительно адаптированная с помощью LoRA-адаптера под специализированную задачу угадывания паролей.

По итогам эксперимента было установлено, что без эволюции промпта, при использовании начальной инструкции, качество на итоговой выборке составило **3.3%**, тогда как после эволюции промпта оно выросло до **7.7%**. Таким образом, улучшение составило **4.4 процентного пункта**, а относительный прирост качества оказался более чем двукратным.

## 2. Использованный стек технологий

Работа выполнялась в среде Kaggle с использованием GPU. 
В качестве базовой модели для предсказания использовалась **Qwen2.5-0.5B-Instruct**, а в качестве прикладного адаптера — **LoRA checkpoint `126_csdn_disQwen0.5B`**. Такая комбинация позволяла использовать относительно компактную модель, уже адаптированную под задачу работы с паролями.

Раота основывалась на двух основыных фреймворках PassLLM и OpenEvolve

## 3. Организация эксперимента

Работа состояла из двух основных этапов.
Первый этам состял из эволюции промта. использовался начльный, вручную написанный промт, и с помощью openevolve промт эволюционировался.
1. бралась текущая версия промпта;
2. модель-мутагенератор предлагала его новую вариацию;
3. новая инструкция передавалась в evaluator;
4. evaluator прогонял её через тот же PassLLM inference pipeline;
5. качество нового промпта измерялось и использовалось для отбора лучших решений.

Важно, что в ходе эволюции менялся только текст инструкции. Архитектура модели, LoRA-адаптер, параметры beam search и остальной inference pipeline оставались неизменными. Это позволило изолировать влияние именно prompt engineering.

Второй этам состял из предсказывания 100 паролей для каждого пароля из тестового набора. Для этого использовался 
beam search и предтренерованная модель на основе qwen2.5 0.5b.
На вход подавался эволюционированный промты, полученный на предыдущем этапе, и старый пароль.

## 4. Как валидировались промпты

Валидация промптов проводилась в два уровня.

### 4.1. Локальная A/B-валидация

Для оперативной проверки кандидатов использовался отдельный режим A/B test. В нём несколько инструкций сравнивались на подмножестве размеченных примеров из `train.json`. Для каждого промпта вычислялись следующие метрики:

- **hit@1** — доля примеров, в которых правильный пароль оказался на первом месте;
- **hit@10** — доля примеров, в которых правильный пароль попал в топ-10;
- **hit@100** — доля примеров, в которых правильный пароль попал в топ-100;
- **mrr@100** — средняя обратная позиция правильного ответа, отражающая не только факт попадания, но и место в ранжировании;
- **top1_is_123456789_rate** — дополнительная диагностическая метрика, показывающая, как часто модель коллапсирует в слишком частотный и нерелевантный пароль `123456789`.

Такой набор метрик использовался не случайно. Одного лишь `hit@100` было недостаточно, потому что некоторые промпты могли повышать recall, но при этом становиться менее устойчивыми и чаще скатываться к популярным универсальным паролям. Поэтому оценка включала и метрики качества ранжирования, и отдельную проверку на нежелательный коллапс.

### 4.2. Оценка в процессе эволюции

Внутри OpenEvolve использовался evaluator, который запускал тот же inference pipeline, что и при основном предсказании. В качестве основной целевой метрики применялась **cracked_rate_at_100**, то есть доля случаев, когда правильный пароль находился среди первых ста кандидатов. Эта метрика использовалась как `combined_score` для сравнения промптов между собой.

## 5. Основные результаты

Исходный промпт представлял собой простую базовую инструкцию, описывающую задачу модели как угадывание пароля по предоставленной информации об аккаунте. Этот вариант можно считать разумной отправной точкой, однако он не был специально адаптирован под особенности конкретного inference pipeline.

После запуска процедуры эволюции был найден более эффективный промпт. Финальная проверка показала следующие результаты:

- **без эволюции промпта**: качество на итоговой выборке составило **3.3%**;
- **после эволюции промпта**: качество на итоговой выборке составило **7.7%**.

Это означает, что эволюция инструкции позволила заметно повысить прикладную полезность модели без изменения её весов. В абсолютных величинах прирост составил **4.4 процентного пункта**, а в относительных — примерно **в 2.3 раза** по сравнению с базовым вариантом.

Такой результат показывает, что для задач данного типа текстовая инструкция действительно влияет на поведение модели. Даже при наличии специализированного LoRA-адаптера способ представления входных данных и формулировка задачи оказываются важными факторами качества.

## 6. Основные проблемы

Основной проблемой являлась нехватка ресурсов kaggle. так как использовалася beam search для поиска кандидатов и всего нужно было сделать предсказания для 8000 паролей, приходилось ужимать BEAM_WIDTH_LIST, EOS_THRESHOLD и BATCH_SIZE чтобы уложиться в рамки 1-2 секунд для генерации списка кандидатов для одного пароля, что ухудшило итоговый результат. 

Так же выявилась проблема с длинной промта. Например, промт данный в статье, давал пердсказание 100 паролей за 5 секунд, из-за чего испольование было затрднителльным, поэтому на эволюционированный промт дополнительно накладывалось ограниечение на его длину.

## 7. Выводы

В ходе работы был построен и протестирован полный пайплайн для исследования prompt engineering в задаче targeted password guessing. Был использован современный стек на основе PyTorch, Transformers, PEFT, PassLLM, OpenEvolve и Ollama. Для валидации промптов применялась как локальная A/B-проверка по нескольким метрикам, так и встроенный evaluator в OpenEvolve, измеряющий фактическое качество угадывания на размеченных примерах.

Главный вывод заключается в том, что **эволюция промпта дала значимый прирост качества**. Если исходная инструкция обеспечивала результат **3.3%**, то после автоматического поиска более удачной формулировки качество на итоговой выборке выросло до **7.7%**. Это подтверждает, что prompt optimization является эффективным инструментом даже в тех случаях, когда сама модель уже зафиксирована и недоступна для дальнейшего обучения.
